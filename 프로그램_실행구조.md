# 미국-한국 주식 연계 자동매매 시스템 — 실행 구조 정리

작성일: 2026-04-02

---

## 1. 전체 개요

미국 증시(선행지표)와 한국 증시(후행지표)의 상관관계를 분석해
시장 국면을 자동 판단하고, 전략 가중치를 조정하며 모의/실전 자동매매를 수행하는
**멀티에이전트 시스템** (Python 3.11, 비동기 asyncio 기반).

---

## 2. 진입점: main.py

```
python main.py [--mode once|schedule|update-history|fetch-history] [--interval 분] [--log-level 레벨]
```

| 모드 | 동작 |
|------|------|
| once (기본) | 파이프라인 1회 실행 후 종료 |
| schedule | 지정 주기(기본 30분)로 파이프라인 반복 실행 |
| update-history | 최근 30일 히스토리 데이터 재수집 후 종료 |
| fetch-history | 5년치 히스토리 데이터 전체 재수집 후 종료 |

main.py 흐름:
1. 로거 설정 (Windows UTF-8 인코딩 처리 포함)
2. argparse로 실행 모드 결정
3. history 모드면 → data/history/fetch_history.py 서브프로세스 실행 후 종료
4. 나머지 모드 → Orchestrator 인스턴스 생성 → run_once() 또는 run_scheduled() 호출

## 3. 핵심 파이프라인 (orchestrator.py)

Orchestrator는 6단계 파이프라인을 순서대로 실행한다.
각 단계 실패 시 graceful degradation (가능한 한 다음 단계 진행).

```
[Step 1] DataCollector.run()
    │  미국/한국/원자재/뉴스 데이터 수집
    │  → 수집 데이터 DB 스냅샷 저장 (save_market_snapshot)
    ↓
[Step 2] Preprocessor.run(step1_result)
    │  표준 데이터폼 변환, 유효성 검증, 이상값 처리
    ↓
[Step 3] asyncio.gather() — 병렬 실행
    ├── MarketAnalyzer.run(step2_result)   → 시장 국면 감지 + 추세 예측
    └── IssueManager.run(step2_result)    → 뉴스 기반 이슈 분류 및 감지
    │
    │  [Step 3-b] 포지션 기간 재평가
    │    WeightAdjuster._position_manager.review_positions_for_horizon_change()
    │    → 국면 변화에 따라 보유 포지션의 holding_period 업그레이드/다운그레이드
    ↓
[Step 4] WeightAdjuster.run(_CombinedInput(step3, issue))
    │  MarketAnalyzer + IssueManager 결과를 합산해 전략 비중 자동 재배분
    │  → SIGNAL 페이로드 생성 (direction: BUY|HOLD, targets, weight_config)
    ↓
[Step 5] LogicApplier.run(step4_result)
    │  전략 라이브러리에서 현재 국면에 맞는 전략 카드 선택
    │  → 진입 조건 정제, strategy_id 결정
    │  (실패 시 step4_result 원본으로 fallback)
    ↓
[Step 6] Executor.run(step5_result)
    │  KIS API로 실제 주문 실행 (MOCK | REAL 모드)
    │  → 손절/익절 관리, 텔레그램 알림
    │  → DB 저장: save_market_phase, save_agent_log, save_account_summary
    ↓
[반환] {status, phase, direction, orders, signal, error}
```

## 4. schedule 모드 추가 동작 (run_scheduled)

```
run_scheduled(interval_minutes=30)
    │
    ├── Debugger.start_background(check_interval=30)   ← 백그라운드 30초 감시 시작
    ├── _stop_take_loop() 태스크 시작                    ← 독립 청산 체크 루프
    │
    └── 무한 루프:
          ├── _update_history_if_needed()  → 날짜 바뀌면 fetch_history.py 자동 실행
          ├── run_once()                   → 6단계 파이프라인 실행
          ├── (6회마다) SystemManager.execute() → 시스템 건강 보고
          ├── debugger.halt_trading 체크   → True면 자동매매 전체 중단
          └── asyncio.sleep(interval_sec) → 다음 실행 대기
```

### 독립 청산 루프 (_stop_take_loop)
- 파이프라인과 별도로 백그라운드에서 보유 포지션 청산 조건을 지속 체크
- 국면에 따라 체크 주기가 달라짐:
  - 변동폭큰 / 하락장 / 대폭락장: **1분** 주기
  - 그 외 안정 국면: **3분** 주기

## 5. 에이전트 구조 (11개)

### 공통 기반: BaseAgent (agents/base_agent.py)

모든 에이전트가 상속하는 추상 클래스. 다음 공통 기능을 제공한다:
- `run()`: timeout + 재시도 래퍼 (asyncio.wait_for 적용)
- `execute()`: 각 에이전트가 구현하는 비즈니스 로직 (abstract)
- `create_message()`: StandardMessage 생성 헬퍼
- `create_error()`: 에러 StandardMessage 생성
- `send_heartbeat()`: 생존 신호 HEARTBEAT 생성
- `_db_log()`: Supabase agent_logs 저장
- `log()`: 로거 출력 (prefix: [에이전트명])

| 레이어 | 코드 | 파일 | 역할 | timeout | 재시도 |

## 6. 통신 프로토콜 (protocol/protocol.py)

모든 에이전트는 **StandardMessage** 객체를 통해서만 데이터를 주고받는다.

```
StandardMessage
├── header (MessageHeader)
│     ├── msg_id       → {agent_code}_{YYYYMMDD}_{HHMMSS}_{순번:04d}
│     ├── version      → "1.0"
│     ├── from_agent   → 송신 에이전트 코드
│     ├── to_agent     → 수신 에이전트 코드
│     ├── timestamp    → ISO 8601
│     ├── priority     → CRITICAL | HIGH | NORMAL | LOW
│     └── msg_type     → DATA | SIGNAL | ORDER | RESPONSE | ERROR | HEARTBEAT | COMMAND | ALERT
├── body
│     ├── data_type    → 페이로드 타입명
│     └── payload      → 실제 데이터 (dict)
└── status
      ├── code         → OK | ERROR | TIMEOUT | RETRY
      └── message      → 상태 메시지
```

### 페이로드 타입 (protocol.py 정의 dataclass)

| 타입 | dataclass | 주요 필드 |
|------|-----------|----------|
| US_MARKET | USMarketPayload | nasdaq, sox, sp500, vix, usd_krw, futures, individual |
| KR_MARKET | KRMarketPayload | kospi, kosdaq, foreign_net, institution_net, stocks |
| COMMODITY | CommodityPayload | wti, gold, copper, lithium |
| MARKET_PHASE | MarketPhasePayload | phase, confidence, elapsed_days, forecast |
| SIGNAL | (dict) | direction, targets, weight_config, phase, reason |
| ORDER | (dict) | order_id, action, code, quantity, price_type, mode |
| ERROR | ErrorPayload | error_id, level, from_agent, error_code, message |
| HEARTBEAT | (dict) | agent_code, status: ALIVE |

## 11. 오류 처리 4단계

```
1. 발생 에이전트
   → ERROR 메시지 생성
   → BaseAgent.run()이 최대 3회 재시도
   → 모두 실패 → ERROR StandardMessage 반환

2. Debugger (백그라운드 감시)
   → 30초마다 에이전트 heartbeat 확인
   → CRITICAL 오류 감지 시 → halt_trading = True → 전체 매매 중단

3. SystemManager (3시간마다)
   → agent_logs 분석, 원인 파악
   → 자동 수정 시도 / 권고사항 생성

4. Orchestrator
   → halt_trading 플래그 감지 → 스케줄 루프 종료
   → run_once() 내 예외 → graceful degradation 또는 에러 반환
   → 텔레그램 알림 (Executor 경유)
```

## 13. 핵심 설계 원칙 요약

1. **표준폼 통신**: 모든 에이전트 간 데이터는 반드시 StandardMessage로만 전달
2. **전략/코드 분리**: 비즈니스 로직은 strategy_config.json에, 코드는 안정적으로 유지
3. **Graceful Degradation**: 각 단계 실패 시에도 다음 단계를 최대한 진행
4. **독립 감시**: Debugger는 어떤 에이전트의 지시도 받지 않고 독립적으로 감시
5. **모의 먼저**: MOCK 모드 검증 후 REAL 모드 전환 (Executor의 mode 필드로 제어)
6. **과최적화 방지**: 3개 이상 구간에서 55% 이상 승률 + 조건 5개 이하 전략만 채택
